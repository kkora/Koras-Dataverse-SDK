# Feature Planning — KDV-005 Batch Operations

> Planning-level design document (pre-implementation). Catalog entry:
> [`feature-catalog.md`](feature-catalog.md#kdv-005--batch-operations-batch).
> Source of truth: [`docs/planning/master-plan.md`](../planning/master-plan.md) §3 (KDV-005),
> §4, §5. Release classification: **MVP**.

## Overview

KDV-005 wraps the Dataverse `$batch` endpoint: multipart MIME payload generation, atomic
change sets, continue-on-error execution, per-item result parsing, and a client-side
1000-operation guard. Batch is the workhorse of data migration jobs (UC-4) and the mechanic
most hand-rolled clients get wrong.

## Requirements

**Functional**

1. Compose a `BatchRequest` from `BatchOperation` items (create/update/upsert/delete —
   operation kinds mirror KDV-002's surface; read operations in batches are a design option,
   subject to implementation).
2. **Atomic change sets:** operations grouped into a change set succeed or fail as a unit,
   matching Dataverse semantics.
3. **Continue-on-error:** when enabled, item failures do not abort the remaining operations;
   per-item outcomes are reported.
4. **Per-item results:** `BatchResponse` exposes an ordered collection of `BatchItemResult`,
   one per operation, carrying success/failure, created ids where applicable, and on failure a
   `DataverseError` (KDV-009).
5. **1000-operation guard:** batches exceeding the documented Dataverse limit are rejected
   client-side before any I/O (master plan §3).
6. Content-ID referencing inside change sets (using the id of an earlier operation in a later
   one) is a design option evaluated during implementation; if excluded from MVP, it is
   documented explicitly.

**Nonfunctional.** Deterministic payload generation (testable via snapshots with pinned
boundaries); bounded memory for maximum-size batches; full cancellation support.

## Proposed public API

Types fixed by master plan §4: `BatchRequest`, `BatchOperation`, `BatchResponse`,
`BatchItemResult` in `Koras.Dataverse.Batches`.

Conservative proposal, subject to implementation:

```csharp
var batch = new BatchRequest();
batch.Add(BatchOperation.Create(new Entity("account") { ["name"] = "Contoso" }));
batch.Add(BatchOperation.Update(existingAccount));

BatchResponse response = await dataverse.ExecuteBatchAsync(batch, ct);
foreach (BatchItemResult item in response.Items)
{
    if (!item.Succeeded) { /* item.Error is a DataverseError */ }
}
```

Change-set grouping and the continue-on-error flag are expressed on `BatchRequest`
(factory/collection shape subject to implementation). Entities and operation payloads reuse
KDV-002's `Entity`/`EntityReference` model unchanged.

## Configuration

- Continue-on-error: per-request flag on `BatchRequest` (default: off — atomic-leaning
  defaults are safer; subject to implementation).
- No global batch options planned; the 1000-op limit is a platform constant, not a knob.

## Error conditions

| Condition | Behavior |
|---|---|
| More than 1000 operations | Client-side rejection before I/O with explicit message |
| Change-set member fails | Entire change set reported failed per Dataverse atomicity; results reflect rollback |
| Item failure with continue-on-error | Remaining items execute; failed item carries `DataverseError` |
| Whole-batch transport failure (429/5xx on the `$batch` call) | Retried per KDV-008; post-retry failure surfaces as `DataverseException` |
| Response/operation count mismatch or unparsable part | Classified error; no silent partial results (subject to implementation detail) |
| Cancellation | `OperationCanceledException` unwrapped; no result object returned |

Design rule: item-level failures are **data** (`BatchItemResult.Error`), not exceptions;
batch-level failures are exceptions. This keeps migration loops allocation- and
control-flow-sane.

## Security

- MIME boundaries are generated by the SDK, never caller-supplied.
- Item payloads inherit KDV-002 serialization safety; item URLs are built with the same
  encoding rules.
- Batch responses are parsed with the same non-polymorphic deserialization constraints
  (master plan §7).

## Performance

- Payload serialization streams parts where practical; maximum-size batches must not require
  proportional intermediate string materialization.
- Response parsing is incremental (part by part).
- Docs guide batch sizing against service protection limits (KDV-008 interplay): bigger is not
  always faster under throttling.

## Observability

- One `Activity` per batch execution, tagged with item count, change-set count, and
  continue-on-error flag (KDV-011).
- Metrics: batch counter, batch duration histogram, failed-item counter tagged by error
  category.
- Logs: batch summary (counts, outcome) at information on failure only; per-item detail at
  debug.

## Test plan

**Unit** (fake `HttpMessageHandler`, payload snapshots with pinned boundary values):
- Payload generation: single ops, change sets, mixed batches, header correctness per part.
- Guard: 1000 accepted, 1001 rejected, empty batch rejected (or no-op — decided and tested
  either way).
- Response parsing: all-success, all-fail, mixed continue-on-error, change-set rollback
  reporting, ordered 1:1 alignment of results to operations.
- Malformed response handling; cancellation before/during send.

**Integration** (env-var gated, master plan §6): mixed create/update/delete batch round-trip;
atomic change set with a deliberately failing member leaves no partial writes;
continue-on-error batch reports the failing item and commits the rest.

## Acceptance criteria

1. Per-item results align one-to-one, in order, with submitted operations.
2. Atomic change sets exhibit verified all-or-nothing behavior against a real environment.
3. Continue-on-error commits successes and reports failures as `BatchItemResult` data.
4. The 1000-operation guard triggers exactly at the limit, before any network I/O.
5. Item errors carry full KDV-009 `DataverseError` fields including request-id data where the
   server provides it.
6. Error, cancellation, and parsing tests present per the MVP bar.
